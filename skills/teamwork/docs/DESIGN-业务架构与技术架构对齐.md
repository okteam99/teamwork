# 设计方案：业务架构与技术架构对齐

> 解决 teamwork skill 中「子项目 = 业务域边界」假设导致的三个对不齐问题。
> 状态：✅ 已落地 v3 · 变更已同步到 SKILL.md / ROLES.md / RULES.md / TEMPLATES.md

---

## 一、问题诊断

### 当前框架的核心假设

```
执行线（业务目标）  →  子项目（sub-project）  →  Feature（可执行任务）
```

这条映射链隐含了一个前提：**每个子项目既是业务边界也是技术边界**。现实中这两个边界经常不重合。

### 三个具体问题

| # | 问题 | 现有机制 | 缺口 |
|---|------|----------|------|
| 1 | 共享技术模块（SDK/中间件/Design System）不属于任何业务子项目 | 无 | 无归属 → 无 ROADMAP → 无文档追踪 |
| 2 | 一个业务 Feature 的代码改动横跨多个子项目 | 跨子项目需求拆分 + DEPENDENCY-REQUESTS.md | 拆分后各子 Feature 独立走完整流程，缺乏「同一业务目标」的关联追踪 |
| 3 | 纯技术工作（重构/升级/基础设施改造）无流程入口 | 四种流程均为业务驱动 | 技术团队只能伪装成 Feature 或不走流程 |

---

## 二、设计原则

1. **最小侵入**：不重构现有四种流程，通过扩展子项目类型和 Feature 元数据解决
2. **向后兼容**：现有单子项目、纯业务项目不受影响
3. **渐进采用**：三个解法可独立启用，不要求一次性全部落地
4. **Single Source of Truth 不变**：STATUS.md / ROADMAP.md / teamwork_space.md 的权威地位不变
5. **PM 主导不变**：所有子项目类型均由 PM 驱动需求，保持角色分工一致性

---

## 三、解法设计

### 解法 A：中台型子项目（解决问题 1 + 3）

#### 核心思路

在 teamwork_space.md 的子项目清单中引入「项目类型」字段，区分两类子项目：

| 类型 | 服务对象 | 典型示例 | PM 视角 |
|------|----------|----------|---------|
| `business`（默认） | 终端用户 | AUTH、WEB、ADMIN | PRD 面向终端用户需求 |
| `midplatform`（中台） | 内部消费方（其他子项目） | shared-utils、infra、design-system | PRD 面向消费方共性需求 |

#### 关键设计决策：PM 仍然主导

对应现实中的中台团队——中台同样有 PM，只是他的「用户」是其他业务线而非终端消费者。中台 PM 的核心能力是：

```
中台 PM 职责：
├── 收集多条业务线对共享能力的需求（消费方需求）
├── 抽象共性，设计通用接口/能力（避免为单一业务线定制）
├── 在 PRD 中定义：消费方是谁、能力边界、API 契约、兼容性承诺
├── 协调消费方的接入节奏（谁先用、谁后用）
└── 确保中台能力演进不破坏现有消费方
```

#### 与现有流程的关系——完全不变

```
中台子项目 vs 业务子项目的流程对比：

                    business（现有）              midplatform（中台）
─────────────────────────────────────────────────────────────
PM 写 PRD          ✅ 面向终端用户                ✅ 面向消费方（其他子项目）
PL-PM Teams 讨论   ✅ 对齐产品方向                ✅ 对齐多业务线共性 + 通用性
PRD 多角色评审      ✅ 全角色                      ✅ 全角色（RD 重点审 API 契约）
Designer 阶段      ✅（如需 UI）                  ✅（如需，如 Design System）
QA → 开发 → 测试   完全相同                       完全相同
完整 Feature 流程   完全相同                       完全相同
```

🔴 **不需要新的文档模板、不需要新的流程、不需要新的角色**。唯一的差异是 PMO 在分析需求时能正确识别「这个需求应该路由到中台子项目」。

#### 纯技术工作的入口（问题 3）

纯技术工作在中台模型下有自然归属——它们就是中台子项目的 Feature：

```
场景 1：框架升级
用户输入：/teamwork 升级 React 到 v19
    ↓
PMO 识别为 Feature 类型，目标子项目：INFRA（中台型）
    ↓
PM 编写 PRD：
├── 消费方：WEB、ADMIN（所有前端子项目）
├── 需求背景：React 18 EOL，安全补丁停更
├── 能力描述：升级到 React 19，保持 API 兼容
├── 消费方影响：各子项目需要回归测试
├── 验收标准：全部消费方构建通过 + 无功能回归
    ↓
后续流程照常（PL-PM 讨论 → 评审 → TC → 开发 → 测试）

场景 2：共享 SDK 新增能力
用户输入：/teamwork 支付 SDK 增加微信支付
    ↓
PMO 识别为 Feature 类型，目标子项目：COMMON（中台型）
    ↓
PM 编写 PRD：
├── 消费方：WEB（微信支付页面）、AUTH（支付鉴权）
├── 能力描述：新增微信支付接口封装
├── API 契约：输入/输出/错误码定义
├── 兼容性：不影响已有支付宝接口
    ↓
后续流程照常
```

中台子项目的 PRD 在标准模板基础上增加以下章节（可选，PMO 提示 PM 补充）：

```markdown

|-------------|-----------|-----------|----------|

| AUTH | 支付鉴权校验 | P0 | 待开发 |

### API 契约（如适用）
<!-- 中台对外暴露的接口定义，消费方依据此开发 -->

### 兼容性承诺
<!-- 对现有消费方的兼容性保证 -->

### 消费方接入计划
<!-- 各消费方何时开始接入、是否需要同步改动 -->
```

#### teamwork_space.md 子项目清单变更

```markdown
## 子项目清单

| 缩写 | 名称 | 类型 | 技术栈 | 职责范围 | 消费方 | 详情 |
|------|------|------|--------|----------|--------|------|
| AUTH | 认证服务 | business | Node.js | 用户认证和权限 | - | [PROJECT.md](...) |
| WEB | 前端应用 | business | React | 用户界面 | - | [PROJECT.md](...) |
| COMMON | 共享库 | midplatform | TypeScript | 通用工具和类型定义 | AUTH, WEB, ADMIN | [PROJECT.md](...) |

```

#### PMO 行为变更

```
PMO 在以下环节感知子项目类型：
├── 需求路由：用户提技术需求时，PMO 能识别应路由到中台子项目
├── PRD 提示：中台子项目 Feature 时，提示 PM 补充「消费方分析」章节
├── PL-PM 讨论：PL 额外关注「通用性」和「对消费方的影响」
├── 完成报告：中台 Feature 完成时，PMO 提醒是否需要通知消费方子项目
└── 其他环节：完全不变
```

| ROLES.md | PM 职责增加中台子项目时的「消费方分析」提示 | 低 |
| RULES.md | 无变更（流程不变） | 无 |

| agents/ | 无变更（所有 Subagent 照常工作） | 无 |

### 解法 B：Feature 业务关联（解决问题 2）

#### 核心思路

在跨子项目需求拆分时，为各子项目的 Feature 建立「业务关联 ID」，使它们可以被追溯到同一个业务目标。

#### 关联机制

```
用户需求：「新增微信支付」
    ↓
PMO 跨子项目拆分：
├── COMMON-F002-支付SDK封装   ← business_goal: "BG-001-微信支付"
├── AUTH-F003-微信支付鉴权    ← business_goal: "BG-001-微信支付"
├── WEB-F004-微信支付页面     ← business_goal: "BG-001-微信支付"
└── 推进顺序：COMMON → AUTH → WEB（中台先行，消费方后接入）
```

#### 与解法 A 的联动

中台模型自然支持「中台先行」的推进顺序——先在中台子项目完成通用能力，再让各消费方接入：

```
PMO 跨子项目拆分时的推进顺序判断：
├── 涉及中台子项目 → 中台子项目的 Feature 优先排序（提供能力的先做）
├── 仅涉及业务子项目 → 按依赖关系排序（被依赖方优先）
└── 用户可覆盖推进顺序
```

#### STATUS.md 扩展

```markdown
## Feature 状态

| 字段 | 值 |
|------|---|
| 当前阶段 | RD 技术方案 |
| 当前角色 | RD |
| 最后更新 | 2026-03-22 |
| 业务关联 | BG-001-微信支付 |           ← 新增（跨子项目时 PMO 自动填写）
| 关联 Feature | AUTH-F003, WEB-F004 |  ← 新增（同一业务目标的其他 Feature）
```

> 单子项目 Feature 这两个字段为空或省略，不影响现有行为。

#### teamwork_space.md 跨项目追踪表扩展

现有的「跨项目需求追踪表」增加 business_goal 字段：

```markdown
## 跨项目需求追踪

| 业务目标 ID | 描述 | 涉及子项目 | 各子项目 Feature | 整体状态 | 推进顺序 |
|------------|------|-----------|----------------|----------|----------|
| BG-001 | 微信支付 | COMMON, AUTH, WEB | COMMON-F002 ✅, AUTH-F003 开发中, WEB-F004 待开始 | AUTH 阶段 | COMMON→AUTH→WEB |
```

#### PMO 行为变更

```
跨子项目 Feature 推进中，PMO 额外执行：
├── 每个子项目 Feature 的 STATUS.md 更新时 → 同步更新 teamwork_space.md 跨项目追踪表
├── 切换到下一个子项目时 → 摘要中显示业务目标整体进度
│   「BG-001 微信支付：COMMON ✅ → AUTH 进行中（当前）→ WEB 待开始」
├── 全部完成时 → 按业务目标输出整体完成报告（而非按子项目）
├── Feature 看板 init 时 → 业务关联 Feature 聚合显示
└── 中台 Feature 完成时 → 提醒下游消费方子项目可以开始接入
```

#### 业务关联 ID 编号规则

```
BG-{三位数字}-{简述}
├── 全局唯一，在 teamwork_space.md 跨项目追踪表中维护
├── PMO 在跨子项目拆分时自动分配
├── 单子项目 Feature 不需要 business_goal（字段省略）
└── 示例：BG-001-微信支付、BG-002-国际化、BG-003-React升级
```

#### 对现有文件的影响

| 文件 | 变更 | 侵入度 |
|------|------|--------|
| SKILL.md | 跨子项目拆分摘要增加 business_goal 分配；Feature 看板增加关联聚合 | 低 |
| RULES.md | 跨子项目推进规则增加业务关联追踪和整体进度报告；中台优先排序规则 | 低 |
| TEMPLATES.md | STATUS.md 模板增加「业务关联」「关联 Feature」可选字段；teamwork_space.md 跨项目追踪表增加 business_goal 列 | 低 |

### 解法 C：依赖发现自动关联（解法 A + B 的联动）

#### 核心思路

当 business 子项目的 Feature 在技术方案阶段发现需要中台子项目提供新能力时，PMO 建议将依赖升级为关联 Feature：

```
WEB-F004（微信支付页面）→ RD 技术方案中发现需要 COMMON 新增支付 SDK
    ↓
PMO 检测到跨子项目依赖
    ↓
PMO 暂停，向用户建议：
├── 选项 1：在 COMMON 创建关联 Feature（COMMON-F002），建立 BG 关联
│   └── 推进顺序调整：暂停 WEB-F004 → 先完成 COMMON-F002 → 回到 WEB-F004
├── 选项 2：写入 DEPENDENCY-REQUESTS.md，等中台团队自行安排
│   └── WEB-F004 继续其他不依赖中台的部分
└── 选项 3：当前 Feature 范围内直接实现（不走中台，但需说明理由）
    └── PMO 记录为技术债务
```

#### 与现有变更管理的关系

这个场景已部分覆盖在 SKILL.md 的「执行中发现的需求边界问题」（line 311-313）。解法 C 将其细化为结构化处理流程，而非依赖 PMO 临场判断。

#### 对现有文件的影响

| 文件 | 变更 | 侵入度 |
|------|------|--------|
| RULES.md | 「执行中发现跨项目依赖」的处理规则细化 | 低 |
| SKILL.md | 变更管理章节增加中台依赖发现的标准处理流程 | 低 |

                    解法 A                    解法 B                    解法 C

────────────────────────────────────────────────────────────────────────────────

## 五、不变量检查

| 不变量 | 是否受影响 | 说明 |
|--------|-----------|------|
| 四种流程（Feature/Bug/问题排查/Feature Planning） | ❌ 不变 | 中台子项目走完全相同的流程 |
| PM 主导需求 | ❌ 不变 | 中台 PM 服务内部消费方，角色不变 |
| PL-PM Teams 讨论 | ❌ 不变 | 中台场景下 PL 额外关注通用性，但机制不变 |
| STATUS.md 是 Feature 级 SSOT | ❌ 不变 | 新增 2 个可选字段（业务关联/关联 Feature） |
| ROADMAP.md 是子项目级路线图 | ❌ 不变 | 中台子项目也有自己的 ROADMAP |
| PMO 阶段摘要格式 | ⚠️ 微调 | 增加业务关联 Feature 的整体进度行 |
| 暂停条件表 | ❌ 不变 | 跨项目拆分仍需用户确认 |
| Subagent 触发规则 | ❌ 不变 | 所有 Subagent 照常工作 |
| Feature 编号规则 | ❌ 不变 | 各子项目内独立递增 |
| 自动流转/暂停规则 | ❌ 不变 | 无新增暂停点 |
| 文档模板体系 | ❌ 不变 | PRD 增加可选章节，不改模板结构 |

## 六、完整示例：微信支付需求

```
用户输入：/teamwork 新增微信支付功能

PMO 分析：
├── 需求类型：Feature
├── 🔀 跨项目需求：是
├── 涉及子项目：
│   ├── COMMON（中台）：支付 SDK 封装（提供微信支付接口）
│   ├── AUTH（业务）：支付鉴权校验（调用 COMMON SDK）
│   └── WEB（业务）：支付页面（调用 COMMON SDK + AUTH 接口）
├── 业务关联：BG-001-微信支付
├── 推进顺序：COMMON → AUTH → WEB（中台先行，消费方后接入）
└── ⏸️ 请确认拆分方案

用户确认后：

Step 1: COMMON-F002-支付SDK封装
├── PM 写 PRD（消费方分析：AUTH 需要鉴权接口 / WEB 需要前端 SDK）
├── PL-PM 讨论（PL 关注：SDK 是否覆盖未来其他支付方式？通用性够不够？）
├── 评审 → TC → 开发 → 测试 → 完成 ✅
└── PMO 提醒：COMMON-F002 完成，BG-001 下一个：AUTH-F003

Step 2: AUTH-F003-微信支付鉴权
├── PM 写 PRD（引用 COMMON-F002 的 API 契约）
├── 完整流程 → 完成 ✅
└── PMO 提醒：AUTH-F003 完成，BG-001 下一个：WEB-F004

Step 3: WEB-F004-微信支付页面
├── PM 写 PRD（引用 COMMON SDK + AUTH 接口）
├── 完整流程 → 完成 ✅
└── PMO 输出 BG-001 整体完成报告

PMO 整体完成摘要：
├── 业务目标 BG-001「微信支付」✅ 已完成
├── 涉及 3 个子项目，共 3 个 Feature
├── COMMON-F002 ✅ → AUTH-F003 ✅ → WEB-F004 ✅
└── 更新 teamwork_space.md 跨项目追踪表
```

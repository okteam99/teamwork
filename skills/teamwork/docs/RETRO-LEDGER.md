# RETRO-LEDGER(teamwork 框架自省台账)

> **telos**:框架自身的「问题 / 优化点 / 有用的地方 / 反思」**跨项目汇总一处** —— consuming 项目目录只放项目的事(`project-specs/PROCESS-LEDGER.md` 度量流程环节价值);框架自己的演化反思归框架仓。「**有用的地方**」是 CHANGELOG 没有的正反馈维度。
> **三层分工**:`CHANGELOG.md` = 单版变更细节(keep-5 · 归档定期清空 · **易逝**)· `PROCESS-LEDGER.md` = 项目侧流程环节价值(一行一 feature)· 本表 = 框架侧自省蒸馏(一行一版 · **永久**)。
> **写入时机**:每次流程完善发版(minor bump)与 CHANGELOG entry **同 commit** —— 发版三件套:CHANGELOG entry + 本表 1 行 + 版本 bump。🔴 单元格 ≤1 行 · 不复述 CHANGELOG(细节在那边 / git)。
> **消费方**:① 年检 /「teamwork 是否该存在」类审视(直接读表算账)② 任何框架改进 session 动手前读最近 10 行(防重复踩坑 / 重复讨论)。
> **膨胀控制**:超 ~500 行 → 年检时把老行主题化压缩(原文 git 可溯)。

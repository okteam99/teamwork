# E2E-REGISTRY 模板

> 📎 **定位**：子项目所有测试关注信息的 **Single Source of Truth**。
> 📎 **核心原则**：每个 REG case 完全自包含——包含执行步骤、前置条件、外部依赖、mock 策略等全部信息，不依赖任何外部文档（Feature 可变更/归档，REG case 不受影响）。
> 📎 **维护时机**：QA 在 Feature 完成时判断是否晋升为 REG 准出 case；Feature 变更影响已有 REG 时同步更新 case 文件。

## REGISTRY.md（入口概览 — 不含执行细节）

> 位置：`{子项目}/docs/e2e/REGISTRY.md`
> 📎 **用途**：子项目准出 case 的索引，按优先级分层。发版前按优先级选择执行范围（P0 快速冒烟 / P0+P1 标准回归 / 全量回归）。
> 📎 **首次创建**：第一个 Feature 完成且 QA 判定需要注册准出 case 时由 PMO 创建。
> 📎 **注意**：REGISTRY.md 只是索引，**不写执行步骤**。所有执行细节在 `cases/REG-{N}-{名称}.md` 中。

```markdown
# {子项目} E2E 准出 Case Registry

> 通用环境准备（服务地址、外部依赖、mock 规则）见 [ENVIRONMENT.md](./ENVIRONMENT.md)。

## 状态说明

- \`✅ 有效\` — 正常回归，case 文件与当前实现一致
- \`⚠️ 待更新\` — 最近 Feature 变更可能影响此 case，需 QA 复核后更新 case 文件（发版前必须先处理为 ✅）
- \`🗑️ 已废弃\` — 功能下线，保留历史

## 回归优先级定义

- **P0 — 核心业务链路**：挂了则用户完全无法使用产品（如登录、支付、核心数据 CRUD）
- **P1 — 重要功能**：挂了影响用户体验但有替代方案（如搜索、通知、非核心页面）
- **P2 — 辅助/边缘功能**：挂了影响较小（如偏好设置、历史记录、管理后台次要页面）

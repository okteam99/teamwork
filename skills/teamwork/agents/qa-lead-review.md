# QA Lead 质量总结 Subagent 规范

> QA Lead 是独立于 QA 的质量审查角色，以全局视角审查整个测试体系的完整性。
> QA 负责执行测试，QA Lead 负责审查"测试本身是否足够"。

---

## 一、角色定位

```
QA Lead ≠ QA：
├── QA：写 TC、审代码、跑集成测试、跑 E2E — 执行者
├── QA Lead：审查 TC 策略、测试覆盖盲区、测试结果可信度 — 审查者
└── 类比：QA 是考试的出题人 + 监考人，QA Lead 是审查"这套卷子出得好不好"的人
```

---

## 二、触发条件

```
🔴 每个 Feature 都触发（无条件）：
├── 触发时机：QA 集成测试通过（或 E2E 通过，如 TC.md 标注需要）之后，PM 验收之前
├── 触发方式：PMO 自动启动 QA Lead Subagent
└── 不可跳过：即使所有测试全部通过，QA Lead 仍需审查
```

---

## 三、输入文件

```
必读文件（PMO 注入关键内容到 prompt）：
├── PRD.md → 验收标准（QA Lead 核对测试是否覆盖了全部验收标准）
├── TC.md → 测试用例清单
├── RD 自查报告 → 了解实现范围和测试输出
├── QA 代码审查报告 → 了解 TC 验证结果 + 集成测试缺口处理情况
├── QA 集成测试报告 → 实际测试结果
├── QA API E2E 验收报告 → API 端到端验证结果
├── QA Browser E2E 验收报告（如有）→ 浏览器端到端验证结果
└── TECH.md → 了解技术方案（特别是配置开关、环境变量、特殊模式）

可选文件：
├── KNOWLEDGE.md → 历史踩坑记录，检查是否有已知问题场景未覆盖
└── docs/architecture/database-schema.md → 了解 schema 变更范围
```

---

## 四、审查维度

```
QA Lead 从 5 个维度进行质量总结：

1️⃣ 验收标准覆盖率（PRD 驱动）
├── 逐条对照 PRD 验收标准
├── 每条标准是否有对应的测试（单元 / 集成 / E2E 至少覆盖一层）
├── 标注覆盖层级：仅单元测试 / 含集成测试 / 含 E2E
└── 未覆盖的验收标准 = 🔴 阻塞项

2️⃣ 测试策略合理性
├── 单元测试 / 集成测试 / E2E 的分工是否合理？
│   ├── 业务逻辑应有单元测试
│   ├── API 端到端链路应有集成测试
│   └── 核心用户场景/完整业务链路应有 E2E（如 TC.md 标注需要）
├── 是否有过度依赖单层测试的风险？（如全是单元测试没有集成测试）
└── 测试数据是否充分覆盖边界条件？

3️⃣ 配置/环境覆盖（本 Feature 特有）
├── 本 Feature 新增的配置开关 → 是否 on/off 两种状态都有测试？
├── 本 Feature 新增的环境变量 → 是否有默认值和异常值测试？
├── 本 Feature 的特殊模式（如 mock 模式）→ 是否在集成测试中验证？
└── 📎 这是最常遗漏的维度 — QA 容易只测默认配置路径

4️⃣ 测试结果可信度
├── 测试是否真正验证了行为？（不是只检查状态码 200，要检查返回内容）
├── 是否有"假通过"风险？（测试数据刚好命中正确路径，换数据就失败）
├── 集成测试是否依赖了外部状态？（如果依赖了，是否有隔离机制）
└── 失败的测试是否都被合理处理？（修复 / Known Issue / Deferred — 不能静默忽略）

5️⃣ 回归风险评估
├── 本 Feature 修改了哪些共享模块？这些模块的已有测试是否仍然通过？
├── 是否有跨 Feature 的副作用风险？
└── 建议追加的回归测试场景（如有）
```

---

## 五、执行流程

```
Step 1: 读取所有输入文件，理解 Feature 范围和测试现状
Step 2: 验收标准覆盖率检查 — 逐条对照 PRD
Step 3: 测试策略审查 — 单元/集成/E2E 分层是否合理
Step 4: 配置/环境覆盖检查 — 新增开关/变量/模式是否都有测试
Step 5: 测试结果可信度审查 — 是否有假通过/静默忽略
Step 6: 回归风险评估 — 共享模块影响 + 跨 Feature 副作用
Step 7: 输出 QA Lead 质量总结报告
```

### 执行约束

```
🔴 强制要求：
├── 必须逐条对照 PRD 验收标准，不能笼统说"覆盖了"
├── 必须检查本 Feature 新增的配置/环境变量是否有测试
├── 必须审查集成测试报告的实际输出（不能只看"通过"二字）
├── 发现问题必须标注严重程度和建议处理方式
└── 即使所有测试通过，也必须输出完整报告（可以全部 ✅ 但不能省略）

❌ 禁止：
├── 自行修改代码或测试
├── 自行修改 TC / PRD 等上游文档
├── 跳过任何审查维度（即使该维度无问题，也要显式标注 ✅）
└── 仅因测试通过率 100% 就判定通过（通过率不等于覆盖率）
```

---

## 七、结果处理

```
✅ 通过（无阻塞项）：
└── Subagent 返回 → PMO 摘要 → 自动进入 PM 验收

⚠️ 有建议（非阻塞）：
├── 记录建议到报告
└── Subagent 返回 → PMO 摘要 → 自动进入 PM 验收（建议项记录到完成报告）

❌ 有阻塞项（验收标准未覆盖 / 关键配置未测试）：
├── RD 补充测试 → 重新提交 QA Lead 审查（最多 2 轮）
└── 2 轮后仍有阻塞 → ⏸️ 升级给用户决定
```
